// Learn ProjectForge

From an idea to a verified starter repository.

ProjectForge turns a project brief and your engineering conventions into a working Python or TypeScript repository. It coordinates Claude Code, Codex CLI, or Google Antigravity, runs the generated project’s checks, and leaves inspectable evidence behind.

// Start here

Install ProjectForge

ProjectForge supports Python 3.12 and 3.13 on macOS and Linux. Install the current published release through Homebrew or as an isolated Python tool.

Homebrew

The supported tap installs ProjectForge with pinned dependencies. If Homebrew reports that an active source virtual environment shadows the commands, run deactivate, refresh shell command lookup, and test again outside the source checkout.

Terminal
$ brew install schramm2/tap/projectforge
$ projectforge --version

uv or pipx

The PyPI distribution is named matt-projectforge. It installs both executable names.

Terminal
$ uv tool install matt-projectforge
$ projectforge --version

# alternative
$ pipx install matt-projectforge

Current source

Use this route when you want to run the v0.7.1 source checkout.

Terminal
$ git clone https://github.com/Schramm2/projectforge.git
$ cd projectforge
$ uv sync --dev
$ ./forge --version
First-run workspaceRun setup from a dedicated directory such as ~/Projects/forge-playground, not from the ProjectForge source checkout. Save that directory during setup if every scaffold should use it. Use projectforge doctor; doctor is a subcommand, not a root option.
Release boundaryThis page documents ProjectForge v0.7.1. For any installed release, its bundled documentation is the exact contract.
// Setup

Connect an AI provider

A live scaffold needs at least one installed and authenticated provider CLI. Forge never collects your token or account identity; authentication stays inside the provider’s own tool.

ProviderInstallProvider-owned loginForge check
Claude CodeOfficial setup ↗claude auth loginDeterministic status check
Codex CLIOfficial repository ↗codex loginDeterministic status check
Google AntigravityOfficial setup ↗Run agy and follow Google Sign-Inagy models; no inference prompt
Readiness check
$ forge doctor
$ forge doctor --json

One provider reported as ready is enough. Missing optional providers do not block a run. The JSON form is deterministic for automation and exits successfully only when config is valid and at least one provider is verifiably ready.

Antigravity over SSHRun agy, open the displayed authorization URL on a local device, and paste the returned code into the remote terminal. Exit the TUI with /exit, then rerun forge doctor.
// Quickstart

Create your first project

1. Preview with zero provider calls

Preview
$ forge --dry-run \
    --name hello-forge \
    --stack python-cli \
    --description "A tiny greeting CLI" \
    --no-docker \
    --no-open

Review the target, stack, options, routing, model behavior, approval mode, convention sources, warnings, and complete prompt. Add --verbose to display full convention source paths and hashes. Preview does not prove provider readiness or generated-project behavior.

2. Run the same scaffold safely

Live scaffold
$ forge \
    --name hello-forge \
    --stack python-cli \
    --description "A tiny greeting CLI" \
    --no-docker \
    --approval-mode safe \
    --no-open \
    --verify

Before the first provider call, Forge shows the workspace, routing, model behavior, provider invocation estimate, duration range, quota and cost context, execution strategy, and verification limits.

3. Or use the interactive flow

Guided setup
$ forge

The first run opens setup. You can use bundled conventions, import nearby AGENTS.md, CLAUDE.md, or Copilot instructions, or create a reusable profile through a short interview. The project questionnaire keeps reusable conventions separate from the current project brief and lets you review every choice before execution.

Demo mode is on by defaultGenerated startup should not require real service credentials. Your provider CLI still requires its own authentication.
// Recipes

Choose a stack

Use the identifier or any alias with --stack. Recipes guide structure, libraries, commands, and Docker defaults; generated project metadata and .forge/verification.json remain authoritative.

Next.js + Reactnextjs · next · react›

npm · Docker off. App Router project with src/app, src/components, src/lib, public, and agent_docs. Tailwind is included in the common guidance.

Typical checks: npm run build, npm run lint, and npx tsc --noEmit. Supports auth, CI, design templates, and media import.

FastAPIfastapi · api›

uv · Docker on. Layered API, domain, application, infrastructure, shared, unit/integration test, and agent_docs structure.

Common guidance includes Pydantic, asyncpg, structlog, httpx, Ruff, strict MyPy, and pytest. Optional services include PostgreSQL with pgvector, OpenAI, and AWS.

FastAPI + AI/LLMfastapi-ai · ai · llm›

uv · Docker on. FastAPI structure extended with retrieval and extraction use cases, vector persistence, and external LLM/embedding integrations.

Adds guidance for pgvector, OpenAI, tiktoken, rank-bm25, tenacity, and numpy. Uses the FastAPI development command family.

Next.js + FastAPI monorepoboth · fullstack · monorepo›

uv + npm · Docker on. FastAPI backend with a frontend Next.js app, shared domain/application layers, tests, Docker files, and agent docs.

Supports auth, CI, design templates, and media import. Typical local ports are API on 8000 and frontend on the Next.js default.

Python CLIpython-cli · cli · typer›

uv · Docker off. A src-layout package with CLI entry point, pytest suite, pyproject.toml, and agent docs. Common guidance includes Typer, Rich, Questionary, Ruff, strict MyPy, and pytest.

TypeScript packagets-package · npm-package · library›

npm · Docker off. ESM-first package with TypeScript source, public entry point, unit/integration tests, compiled output, strict tsconfig.json, and Vitest.

Python workerpython-worker · worker · service›

uv · Docker on. Service structure with CLI dispatcher, models, storage, business services, config, infrastructure, tests, and agent docs.

Common guidance includes boto3, requests, structlog, dotenv, pytest, and a Docker build. Typical services include DynamoDB, ECS Fargate, EventBridge, and Slack.

All Python recipes broadly favor uv, Ruff, strict MyPy, pytest, and pre-commit. TypeScript recipes favor strict TypeScript, ESM-first setup, and Vitest-oriented testing.

// Reusable guidance

Teach Forge how you work

Conventions are durable engineering rules reused across scaffolds. A project brief describes only the current product. Keeping them separate makes both easier to inspect and maintain.

Precedence

  1. Bundled ProjectForge defaults.
  2. Selected profile at ~/.forge/profiles/<name>.md.
  3. Legacy-compatible ~/.forge/conventions.md.
  4. Project-local .forge/conventions.md.

Later layers have higher precedence. Empty placeholders and known generated legacy mirrors are ignored with a warning. Credential-shaped content is rejected before a provider call.

forge conventions init teamCreate a starter profile without overwriting an existing one.
forge conventions import ./AGENTS.md --name teamImport a Markdown instruction file into a new profile.
forge conventions listList profiles and show the selected profile.
forge conventions select teamUse a profile for future scaffolds.
forge conventions inspect --stack fastapi --jsonShow exact source order, paths, warnings, and SHA-256 hashes.
forge conventions preview --stack fastapiPrint effective content without starting a provider.
forge conventions validate --stack fastapiValidate the selected profile and effective bundle.
forge conventions edit teamOpen a profile in the configured editor.
Private instructionsImports are Markdown-only and size-bounded, but profiles and exported prompts can still contain private organization guidance. Review them before sharing.
// Control the run

Execution modes and routing

safe · default

Provider workspace-write mode with bounded execution controls. Normal scaffolding belongs here.

plan

Read-only provider planning. It still makes provider calls and can consume quota.

unsafe --allow-unsafe

Removes approval or sandbox boundaries. Use only inside external isolation you control.

--dry-run

No project, provider process, or model call. Best first step for every repeatable scaffold.

Safe mode still permits the provider to edit the target workspace, run allowed commands, install dependencies, use allowed network access, and consume quota. Inspect preflight before continuing.

Automatic routing

Forge can route specialist phases across ready providers and falls back in the declared order Claude → Antigravity → Codex. Use --use claude, --use antigravity, or --use codex to force one provider for a run.

Provider defaults control model selection unless you deliberately pass --model. Forge avoids maintaining a volatile model catalog; consult live provider help for accepted values.

Standard versus multi-agent

Standard execution normally uses one provider invocation per phase. --agents decomposes phases into several focused tasks and can improve complex output, but it is slower and typically uses substantially more provider calls. Preflight quantifies the expected invocation range.

// Trust the result

Verification and evidence

A successful provider exit means files were generated. It does not mean the project works. ProjectForge reports Project Ready only when all required checks pass; Python verification includes required handoff files, tracked uv.lock behavior, and bounded console-entry-point smoke checks. Live runs start inside an unborn main repository while local Forge and graph state remain excluded from provider commits. Skipped or incomplete verification yields the honest Project Created.

Generated recordWhat it provesSensitivity
.forge/progress.jsonPrompt hashes, phase status, attempts, duration, failure class, and resume contract.No prompt or provider output.
.forge/scaffold.jsonRequested facts, routing, model behavior, approval mode, and convention/context provenance.Metadata only; inspect before sharing.
.forge/conventions-snapshot.mdExact convention input used for replay.Potentially private.
.forge/context-snapshot.mdApproved project brief and selected nearby content, when supplied.Potentially private.
.forge/verification.jsonCommands, relative working directories, timeouts, exits, health endpoints, and remediation.Redacted durable evidence.

For high-confidence delivery, rerun the recorded project commands independently. Python projects should commit uv.lock, include the required pre-commit configuration, and can declare bounded localhost health settings:

pyproject.toml
[tool.forge.verification]
health_endpoints = ["/healthz", "/readyz"]
health_startup_timeout = 20
health_request_timeout = 4

Forge accepts 1–8 local path-only endpoints, a 1–120 second startup timeout, and a 1–30 second request timeout. Invalid metadata falls back to safe defaults, and host/port remain derived from the generated local run command.

// Preserve completed work

Resume and recover

Do not delete a partial project or pay to rerun completed phases. Fix the classified problem, preserve .forge/progress.json, and repeat the original command with --resume.

Resume
$ forge \
    --name hello-forge \
    --stack python-cli \
    --description "A tiny greeting CLI" \
    --no-docker \
    --approval-mode safe \
    --no-open \
    --verify \
    --resume

Resume validates project name, stack, provider routing, prompt hashes, approval mode, and effective conventions. It skips completed phases and rejects drift instead of mixing two scaffold contracts.

Provider failure

  1. Read the bounded failure category and remediation.
  2. Run forge doctor for install, login, or connectivity readiness.
  3. For an explicit model error, remove --model or choose a provider-supported value.
  4. Repeat the original request with --resume.

Verification failure

Open .forge/verification.json and run the first failed command from its recorded project-relative directory. Fix the generated project; do not regenerate it just to hide a failed check.

// Local settings

Configuration and local data

User-owned settings live under ~/.forge/. Set FORGE_HOME to another directory for isolated automation or ephemeral environments.

User config

~/.forge/config.json is created by forge --setup, schema-validated, atomically written, and restricted to user permissions. Change ordinary preferences by rerunning setup rather than hand-editing JSON.

KeyMeaning
preferred_editorEditor identifier used for optional CLI or macOS application opening.
available_backendsProviders seen during setup; readiness is always rechecked.
backend_modelsOptional advanced model overrides. Empty means provider default.
docker_availableDocker availability observed during setup.
projects_dirDefault parent directory for generated projects.
agentsStandard or multi-agent execution preference.
soundWhether successful scaffolds play a local completion sound.
conventions_profileSelected reusable convention profile.

If config is invalid, Forge moves it to config.json.corrupt-<timestamp> when possible, uses safe defaults, and asks you to rerun setup. Review the recovery copy locally before removing it.

Design templates and media

Frontend-capable stacks can use the bundled default-design-guide, a user override under ~/.forge/design-templates/, or a project override under .forge/design-templates/. Named media collections come from ~/.forge/media/<collection>/, or $FORGE_HOME/media/<collection>/ when overridden; arbitrary paths are not accepted. Move collections from an older source checkout into the user-owned directory before removing that checkout, and do not store assets in a Homebrew Cellar path.

Post-scaffold hook

An executable ~/.forge/hooks/post-scaffold.sh runs after verification with a 60-second limit. It receives FORGE_PROJECT_DIR, FORGE_PROJECT_NAME, FORGE_STACK, and FORGE_DEMO_MODE. It is user-authored shell code running with your account’s permissions—review it before use.

Local history

scaffold.log, quality.jsonl, and preferences.json stay local and help Forge report history, improve routing, and select interactive defaults. Forge does not transmit these analytics.

// Know the boundary

Security and privacy

What leaves your machine

During a live run, Forge sends the assembled project brief, effective conventions, explicitly selected nearby Markdown, design or media context, and phase instructions through the chosen provider CLI. The provider’s terms, retention policy, account settings, and billing apply.

Nearby context discovery checks only known filenames in the current directory and includes no file content until you select it. Selected files are limited to 32 KB each and 64 KB total, then secret-scanned before prompt assembly.

What Forge does not collect

  • Provider credentials, tokens, or account identity.
  • Raw provider output in progress records or durable failure evidence.
  • Absolute target paths in local scaffold history.
  • Local quality and preference analytics.

Your responsibility

  • Never put credentials in descriptions, extra instructions, conventions, selected context, exported prompts, or generated fixtures.
  • Treat convention and context snapshots as potentially private.
  • Review generated code as untrusted until checks pass.
  • Use unsafe mode only inside an isolation boundary you control.
  • Review post-scaffold hooks as normal shell code with your user permissions.
Reporting safelyDo not attach .env content, raw prompts, provider identity, convention snapshots, context snapshots, credentials, or unredacted local output to a public issue.
// Diagnose before retrying

Troubleshooting

forge: command not found

For source installs run uv sync --dev and use ./forge. For uv or pipx inspect uv tool list or pipx list. For Homebrew inspect brew info schramm2/tap/projectforge. If Foundry owns forge, run projectforge.

Homebrew command is shadowed

Leave the active source virtual environment with deactivate, run rehash in zsh or hash -r in bash, then verify command -v projectforge from a directory outside the source checkout. Do not disable Homebrew trust or path checks to hide this warning.

Provider is missing or needs login

Install and authenticate through the provider’s official CLI, then run forge doctor. Forge never asks you to paste a credential.

Antigravity says needs_login

Run agy, complete Google Sign-In, exit with /exit, then rerun doctor. For check_inconclusive, verify connectivity and keyring access and run agy models directly.

Config is corrupted

Forge preserves a timestamped recovery copy when possible. Run forge --setup, then forge doctor. Review the local recovery file before deleting it.

Permission or sandbox failure

Keep safe mode. Confirm the target workspace and adjust only the provider’s scoped policy. Do not use unsafe mode as routine recovery.

Existing target directory

Interactive Forge offers rename, confirmed overwrite, or cancel. Use --resume only when the directory contains matching .forge/progress.json evidence.

Post-scaffold hook fails

Confirm ~/.forge/hooks/post-scaffold.sh exists, is executable, and finishes within 60 seconds. Review it before running manually.

Shell completion is stale

Refresh completion
$ forge --install-completion
// Lifecycle

Upgrade and uninstall

Upgrade a published installation

Upgrade
$ uv tool upgrade matt-projectforge
# or
$ brew upgrade schramm2/tap/projectforge

Migrating older installations

The current source uses projectforge as the canonical Python namespace and preferred command, while retaining forge as a compatibility command. The PyPI distribution remains matt-projectforge.

Legacy 0.4.1 config is normalized in memory. Existing model overrides for supported providers remain; retired Gemini backend selections become Antigravity while incompatible Gemini model overrides are discarded. Run forge doctor and forge --setup after upgrading. Scaffolds created before phase ledgers cannot be resumed retroactively.

Uninstall

Uninstall
$ uv tool uninstall matt-projectforge
# or
$ brew uninstall projectforge

Removing the application does not delete user-owned profiles, configuration, snapshots, or history under ~/.forge/. Back up or move that directory separately if you no longer need it.

Get support

For ordinary bugs, open a GitHub issue ↗ with the ProjectForge and provider versions, operating system, approval mode, minimal reproduction, and redacted evidence.

For a vulnerability, use the repository’s private vulnerability report ↗. Do not open a public issue containing an exploit, credentials, private conventions, or provider account details.
